# 需求文档

## 引言

本规范旨在审查现有代码库实现，评估token消耗业务场景是否已正确实现。token系统用于跟踪和控制平台各种功能的资源使用情况。

## 与产品愿景的契合度

token消耗系统对平台的可持续性和用户体验至关重要。它确保公平的使用跟踪，同时为不同的访问模式和管理控制提供灵活性。

## 需求

### 需求1：SiteRank Token消耗

**用户故事**：作为平台用户，我希望在SiteRank中每次成功查询域名时消耗1个token，无论数据是否来自缓存，这样我就能为我的研究活动预测token消耗。

#### 验收标准

1. 当用户在SiteRank中成功查询域名时，系统应从用户余额中扣除1个token
2. 当SiteRank查询返回缓存数据时，系统仍应扣除1个token
3. 当SiteRank查询失败（超时、错误、无效域名）时，系统不应扣除任何token

### 需求2：BatchOpen简单HTTP访问Token消耗

**用户故事**：作为平台用户，我希望在使用"简单HTTP访问"模式进行BatchOpen时，每个成功访问的URL消耗1个token，这样我就能以最少的token成本高效执行批量URL检查。

#### 验收标准

1. 当用户通过简单HTTP成功访问URL时，系统应从用户余额中扣除1个token
2. 当简单HTTP访问失败（404、超时、连接错误）时，系统不应扣除任何token
3. 当BatchOpen使用简单HTTP模式时，系统应跟踪每个URL的成功/失败状态以进行token扣除

### 需求3：BatchOpen高级浏览器访问Token消耗

**用户故事**：作为平台用户，我希望在使用"高级浏览器访问"模式进行BatchOpen时，每个成功访问的URL消耗2个token，这样我在需要时可以执行完整的浏览器渲染和JavaScript执行。

#### 验收标准

1. 当用户通过高级浏览器成功访问URL时，系统应从用户余额中扣除2个token
2. 当高级浏览器访问失败时，系统不应扣除任何token
3. 当BatchOpen使用高级浏览器模式时，系统应跟踪每个URL的成功/失败状态以进行2个token的扣除

### 需求4：非侵入式Token消耗实现

**用户故事**：作为系统架构师，我希望token消耗规则能在不影响核心业务功能的情况下实现，这样token系统就可以在不中断服务操作的情况下添加或移除。

#### 验收标准

1. 当添加/修改token消耗时，核心业务逻辑应保持不变
2. 当token扣除失败时，主要操作仍应成功完成
3. 当token系统被禁用时，所有功能应继续正常运行

### 需求5：可配置的Token规则支持热更新

**用户故事**：作为管理员，我希望通过GoFly后台管理系统配置和修改token消耗规则，并支持热更新，这样我就可以在不服务停机的情况下调整token经济规则。

#### 验收标准

1. 当管理员在GoFly中更新token规则时，系统应立即应用更改，无需服务重启
2. 当token规则更新时，所有活动会话应对后续操作使用新规则
3. 当GoFly后台不可用时，系统应继续使用最后已知的有效配置

## 非功能性需求

### 代码架构和模块化
- **单一职责原则**：token消耗逻辑应在专用模块中隔离
- **模块化设计**：token跟踪应是与业务操作分离的关注点
- **依赖管理**：核心功能不应依赖于token系统的可用性
- **清晰接口**：token消耗应暴露清晰、定义良好的契约

### 性能
- token扣除操作不应为主要操作增加显著延迟
- token规则缓存应在高频操作期间最小化数据库查找
- 批量操作应高效处理token跟踪

### 安全性
- token消耗不能被客户端请求绕过或操纵
- 所有token交易必须记录以供审计
- 配置更改必须需要适当的身份验证和授权

### 可靠性
- token系统故障不应阻止核心功能
- token余额应保持事务一致性
- 配置更新应是原子的且可回滚的

### 易用性
- token消耗应对最终用户透明
- 当token操作失败时应提供清晰的反馈
- 管理界面应为规则配置提供直观的操作
